The ladder is intentionally two-tier, not the four-tier design originally sketched. True simultaneous-open TCP hole punching was deliberately dropped in favor of UPnP (handles the common cone-NAT case with zero user config) plus a TURN relay (guarantees connectivity through symmetric NAT/CGNAT, which is exactly where hole-punching tends to fail anyway) — a smaller, more reliable surface than a three-tier ladder with a flaky middle rung.

The share code

A share code is a self-contained rendezvous token — no server involved. It base32-encodes version(1) + sender's public IPv4(4) + sender's DHT UDP port(2) + full 256-bit file hash(32) into 39 bytes. The receiver decodes it, bootstraps its own DHT node directly against the sender's DHT address, then discovers the sender's TCP chunk-server address (and any TURN relay info) through the normal DHT announce/get-peers mechanism.

This is why the code is longer than the web app's 4-character room codes: a content-addressed, serverless swarm key needs the full hash to avoid collisions, where the web app's codes are just ephemeral signaling-server lookup keys. See packages/cli/src/lib/shareCode.js.

Phase B

NAT traversal & real-internet connectivity

done

The hardest phase, and the one that decides whether this is a real public tool or a LAN-only toy.

  • UPnP / SSDP. net/upnp.js — zero-dependency SSDP discovery + WANIPConnection SOAP client. mesh send tries it automatically unless --no-upnp.
  • STUN. net/stun.js — RFC 5389 client for public-IP discovery, verified against real Google STUN servers via mesh diagnose.
  • TURN relay, last resort. net/turn.js — a real RFC 5766 client (long-term credential challenge/response, stale-nonce recovery, Allocate/CreatePermission/Send/Data), reusing the exact HMAC-SHA1 credential scheme from packages/signaling/src/server.js.
  • Reliable-datagram shim. net/reliableDatagram.js — a sliding-window ARQ so the TCP-shaped wire protocol runs over TURN's lossy relay. Stress-tested with simulated 15% loss + reordering + duplication.
  • The ladder. net/connect.js — direct dial, and on failure (if the peer advertised relay info) a RELAY_HELLO over the DHT authorizes the connection on the sender's TURN allocation, then a relayed ReliableDatagramChannel takes over — transparently, from PeerConnection's point of view.
  • Deliberately dropped: true UDP hole-punching. Symmetric-NAT hole-punching has a poor success rate anyway; UPnP + TURN cover the cone-NAT and symmetric-NAT/CGNAT cases respectively with a smaller, more reliable surface.
Verified
A full send/receive round trip over a real fake-TURN relay server, plus direct-dial fallback, both covered by automated tests.

Bugs found along the way

Building the NAT layer surfaced several real, pre-existing issues in code that predated this work — not just in the new modules.

Path traversal via remote file name. downloadFileByHash and the (now-removed) legacy receiver.js used the sender-supplied file name as an output path verbatim — a malicious sender could offer a file named ../../../.ssh/authorized_keys. Fixed with path.basename().
chunkServer/DHTNode defaulted to binding 127.0.0.1 only — silently unreachable from any other machine, even on a LAN. Both now default to 0.0.0.0, with a new publicAddress concept on DHTNode so self-announced entries carry something actually connectable instead of the wildcard address.
PeerConnection's close handler never cleared pending chunk-request timeouts. Every connection close left a live 30-second timer running. Found by noticing npm test mysteriously took 30 extra seconds after every subtest had already passed.
ReliableDatagramChannel didn't guard channel.send(). A closed underlying transport throwing would crash the whole process instead of failing just that one channel. Now wrapped to fail gracefully.
Sender-side relay demux was keyed to the wrong port. The first design pre-bound a per-peer relay channel using the port seen on the RELAY_HELLO DHT message — but the receiver talks to the actual TURN relay from a different local socket, which can get a different external port under symmetric NAT. Fixed by deferring channel creation until the first real Data Indication arrives and learning the true port then (createRelayListener() in turn.js).
DHT node identity must not be persisted per machine. An earlier draft cached one DHT node ID in a config file for routing-table stability. Running mesh send and mesh receive on the same machine then gave both processes the same ID, and the routing table silently refuses to add a peer sharing your own ID — breaking discovery between two processes on one computer, a completely normal thing to do. Removed; each process now gets a fresh random ID.

"Will we use TURN?"

Yes — as the fallback tier of a two-tier ladder, not the primary path: direct dial first, TURN relay only when that fails. That mirrors what the browser/WebRTC side already does — STUN/direct first, TURN only when direct candidates fail — and it reuses the same coturn deployment and credential-minting algorithm already running for the web app. No new server infrastructure: a real RFC 5766 TURN client living in the engine, driven by the CLI instead of a browser's WebRTC stack.
