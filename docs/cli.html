In short: the P2P engine is genuinely solid, tested infrastructure. The gap isn't unfinished logic — it's that nothing calls it from a terminal yet, and nothing helps two home routers find each other.

Also worth retiring: packages/engine/receiver.js and sender.js are an older demo pair that predates the DHT+swarm integration. sender.js announces to the DHT, but receiver.js never touches the DHT, swarm, or resume logic — it dials a hardcoded host:port directly. They work, but they duplicate what transfer.js + chunkServer.js already do properly. Delete once the real CLI commands are proven.

Security finding from this audit

Path traversal via file name. Both transfer.js's downloadFileByHash and the legacy receiver.js take the output file name straight from the remote peer's FILE_OFFER message — resolvedOutputPath = outputPath || manifest.fileName. On a trusted LAN demo that's harmless. Once the CLI is talking to arbitrary internet peers over the DHT, a malicious sender can offer a file named ../../../.ssh/authorized_keys and the receiver will happily write attacker-controlled bytes there. Sanitize with path.basename() and reject any path that resolves outside the output directory — this needs to land in Phase A, not be deferred to a later "hardening" pass.

The six phases

Each phase ends at a working, demoable checkpoint — same discipline as the rest of this project's build log.

Phase A

Wire the CLI to the engine

Goal: mesh send ./file.zip and mesh receive <code> work end-to-end on one machine / LAN, using what's already built — no new engine features.

  • Real commander program in src/index.js: mesh send <path> [--port] [--bootstrap host:port], mesh receive <code> [--out dir] [--bootstrap host:port], mesh seed <path>.
  • commands/send.js: index → start DHT node → bootstrap → open chunk server → announce. Print a short share code, not a raw 64-char hash.
  • commands/receive.js: resolve code → bootstrap DHT → downloadFileByHash() → wire SIGINT to an AbortController for graceful pause (resume.js already supports it).
  • Fix the path-traversal bug as part of this phase, not deferred.
  • Retire receiver.js / sender.js once the new commands are proven equivalent.
  • New test: spawn send and receive as real child processes, assert the file arrives byte-identical.
Exit criteria
Two terminals, same machine, mesh send / mesh receive, file arrives correctly, resumes after Ctrl+C.
Phase B

NAT traversal & real-internet connectivity

The hardest phase, and the one that decides whether this is a real public tool or a LAN-only toy.

  • B1 — UPnP / NAT-PMP. On mesh send, try to auto-open a router port mapping (pure-JS lib, no native deps). Falls through silently if unsupported. Cheapest win.
  • B2 — UDP hole punching via the DHT as rendezvous. Add a STUN binding request so each node learns its own reflexive address; exchange it through an already-reachable node, then both sides fire UDP simultaneously to punch through NAT. Needs a UDP sibling transport to the existing TCP PeerConnection/chunkServer — TCP simultaneous-open is much less reliable.
  • B3 — TURN relay, last resort. Reuse the coturn container already in docker-compose.yml, and reimplement the same HMAC-SHA1 time-limited credential scheme from packages/signaling/src/server.js — it's plain TURN auth, nothing WebRTC-specific about it.
  • Ladder: direct dial → after UPnP mapping → UDP hole punch → TURN relay. Log which tier won, surface it in the TUI.
Exit criteria
Two machines on different home networks, no port forwarding, no shared LAN — file transfers successfully.
Phase C

Production CLI UX

  • Share codes instead of raw hashes — e.g. mesh receive SWIFT-EAGLE-42, encoding the fileHash plus a bootstrap hint.
  • Ink TUI (currently a one-line stub): live progress bar, speed, connected-peer list, pause/resume state. Fall back to plain logging when stdout isn't a TTY.
  • Config persistence at ~/.config/mesh/config.json — a stable DHT node identity across runs, plus known-good bootstrap peers.
  • Signal handling — SIGINT → pause + checkpoint, already supported by resume.js, just needs a process.on('SIGINT', …) wire-up.
  • Folder support: chunker is single-file today. Ship single-file first; treat folders as a fast-follow rather than blocking release.
Exit criteria
mesh send feels like a real tool a stranger could use without reading the source.
Phase D

Packaging & npm publish

  • Bundle, don't workspace-link. Build @mesh/engine + commander + ink into one file via esbuild (zero native deps — crypto/net/dgram/fs are Node core) so a global install doesn't need the monorepo.
  • Package name: registry check confirms mesh and mesh-cli are already taken. Free: mesh-p2p, meshdrop, mesh-transfer, meshfile, mesh-dht. The npm package name and the bin command name are independent — you can publish as mesh-p2p and still ship a mesh command. Safest option: scope it under your own npm username, e.g. @yourusername/mesh-cli — guaranteed collision-free.
  • package.json: add files, engines.node >=18, repository, homepage, keywords, publishConfig.access if scoped.
  • Extend .github/workflows (only deploy.yml exists today) with a test-then-publish workflow on version tags.
Exit criteria
npm install -g mesh-p2p on a clean machine, mesh send/mesh receive work, no monorepo in sight.
Phase E

Hardening & security pass

  • Confirm the path-traversal fix landed.
  • Rate-limit / cap simultaneous connections per IP on chunkServer.js — it's internet-facing now, not LAN-only.
  • Validate all CLI-supplied paths, including --out, never resolve outside the intended directory.
  • Cross-platform pass — Windows Ctrl+C/SIGINT behaves differently than POSIX; test UPnP libs and path handling on Windows specifically.
  • Run a full security review once Phases A–D are code-complete, before publishing.
Phase F

Docs, demo & release

  • CLI-specific README.md inside packages/cli — npm renders this on the registry page.
  • Short demo GIF or asciinema of send + receive across two real machines.
  • Update docs/phases.md's Phase 5 line once this lands — it currently understates what's already built.
  • Tag a release, npm publish.

"Will we use TURN?"

Yes — as the last tier of a connection ladder, not the primary path: direct dial → UPnP-assisted direct → UDP hole punch (DHT-mediated rendezvous) → TURN relay. That's exactly what the browser/WebRTC side already does — STUN first, TURN only when direct candidates fail — and it reuses the same coturn deployment and credential-minting algorithm already running for the web app. No new server infrastructure, just a TURN client living in the CLI/engine instead of the browser's WebRTC stack.

Suggested order, and what's next

Phase A is the highest-leverage, lowest-risk work — it makes ~90% of already-built, already-tested code reachable from a terminal. Phase B (NAT traversal) is the hardest and most time-consuming by a wide margin, and it's the phase that decides whether this becomes a genuinely usable public tool. Phases C–F are comparatively mechanical once A and B exist.

Recommended sequencing: A → E (path fix now) → B → C → D → E (rest) → F.
